## Learned User Preferences

- Uses Bun exclusively — not npm, pnpm, or Node; workspace rule enforces this
- Still in planning/prototyping — wants a concise plan before schema/logic changes, and it's ok to rewrite migrations wholesale instead of adding new ones
- v0 product focus is task management + time tracking (Asana + BigTime replacement); forms are not a core table/feature; matter staffing uses `matter_staff`/`matter_role` (not `matter_attorneys`, since paralegals/EAs can be staffed) but staffing workflows can stay later scope
- API connects directly to Postgres via Kysely, not via supabase-js
- Invoice payment allocation modeled as multiple payment rows linked by external_payment_id, not a junction table; `total_amount` is stored and can be overridden via an explicit `adjustment_amount` column, not silently recomputed
- Time entries without a task_id are non-billable, reporting-only (billable/charge_client flags live on tasks, not time_entries); do not store per-entry billable hours on time_entries — task-level billable totals are derived by summing entries for the task and rounding
- Multi-tenant product: firm-specific config (e.g. late_fee_rate) must not have defaults — each firm sets its own values
- Pushes back on premature optimization; prefers query simplicity until scale pain is measured
- Values concise DX — prefers a single well-considered solution (views, triggers, scopedDb, runAs) over belt-and-suspenders safeguards or app-level wrappers that require type hacks; if a workaround seems necessary, stop and present options/recommendations before coding
- Bun type deps only: `@types/bun` with tsconfig `"types": ["bun"]`; never add `@types/node` even when `Buffer` is needed
- Audit trail is a hard requirement — store JSONB diffs only (not full row snapshots); `changed_by` read from `app.acting_user_id` GUC (set by `runAs` / `runAsEmployee`) with row's `updated_by` as fallback, so direct-SQL writes still log
- Prefers zod `satisfies` pattern for request validation — define schemas once and assert they satisfy `Insertable<DB[...]>` / Patch types, rather than duplicating column lists across SQL and TS

## Learned Workspace Facts

- Bun monorepo: `apps/*` (`office-web` — Vite + React, port 5173; `api` — Hono on Bun runtime, port 8000) plus `packages/*` (`@makase-law/types`, `@makase-law/shared`, `@makase-law/utils`); root scripts `dev:api`/`dev:office` (no combined "run all" — two terminals); a `client-portal-web-app` is anticipated but not yet in the repo
- Frontend stack: Vite + React 19 + wouter (routing), SWR (data fetching), shadcn primitives sitting on Base UI (`@base-ui/react`, NOT Radix — `asChild` is the wrong pattern, use Base UI's `render` prop; CSS vars are `--anchor-width` / `--available-height`, not `--radix-*`), Tailwind v4, Supabase JS for auth only — server data goes through `apps/office-web/src/apis/use-api.ts` against `VITE_API_URL` with `Authorization: Bearer <supabase JWT>`
- App DB objects live in the `app` schema; soft-deletable tables use underscored base tables plus `security_invoker = true` active views (e.g. `app._matters` -> `app.matters`) and may expose `_all` views for ergonomic joins, while non-soft-deleted tables can be direct clean tables (e.g. `app.offices`, `app.tasks`, `app.invoices`)
- `INSTEAD OF DELETE` triggers (`app.prevent_delete`) attached to every view to block hard deletes through views; all `app.*` functions pin `SET search_path = ''`; `anon`/`authenticated` roles fully revoked from `app` (public schema reserved for views when frontend/realtime access is needed); Postgres extensions (e.g. `pg_trgm`) live in a dedicated `extensions` schema, never `public`, to satisfy Supabase advisors
- Kysely with `.withSchema("app")` over Bun's built-in SQL driver; Kanel + `kanel-kysely` via `packages/types/kanel.config.cjs` → `packages/types/src/generated`; `@makase-law/types` exports `DB` plus route/service aliases from `makase-types.ts` (e.g. `Offices`, `NewOffices` — `New*`/`*Patch` omit trigger-owned audit and soft-delete columns); `pg` is a devDep for codegen only; in `.where((eb) => ...)`, `eb` is `ExpressionBuilder` — use `eb(column, operator, value)` and `eb.or` / `eb.and`, not `eb.where`
- `packages/shared` (`@makase-law/shared`) exports `_rootDb` (root Kysely pool, only `runAs*` should open transactions on it), `runAsUser` / `runAsEmployee` / `runAsSystem`, scope helpers (`getScope`, `buildMatterBasedScopeFilter` for tables with `matter_id`, `buildNonMatterScopeFilter` for tables without), and the service layer under `services/office-scoped/*` and `services/user-scoped/*`; subpath exports `./context` and `./schemas`; consumed by `apps/api` (and future cron scripts) via `workspace:*`
- Two-stage attribution per request: `authenticateUser` middleware verifies the Supabase JWT then opens `runAsUser(user_id, next)` (sets `app.acting_user_id` GUC, pins a Kysely transaction); `authorizeEmployee` middleware (mounted under `/office/:office_id`) loads employment + merged permissions + team IDs + `custom_matter_access` add/block lists in parallel and wraps `runAsEmployee({ employee, permissions, teamIds, blockMatterIds, addMatterIds }, next)` (sets `app.acting_office_id` GUC). `custom_matter_access` rows override scope per-matter (`add` grants access regardless of team/self scope; `block` revokes it — admins bypass blocks, so conflict-of-interest must be enforced at write-time validation). `set_audit_fields()` BEFORE-trigger on every `_table` writes `created_by`/`updated_by` from the GUC and RAISES if unset — every write must run inside one of these scopes. Auth triggers (`handle_new_auth_user`, `handle_auth_user_updated`) set the GUC to `NEW.id` so auth-originated profile writes self-attribute. `runAsSystem(office_id, fn)` covers bootstrap/cron — attributes to a seeded SYSTEM user (`00000000-0000-0000-0000-000000000001`, display name `'SYSTEM'`), and `getScope` short-circuits to `'office'` for system context. All `runAs*` throw on nested scopes (one actor per request/transaction)
- Named `{table}_id` PK convention (user_id, office_id, matter_id, entity_id, task_id, invoice_id, etc.) — no generic `id`; identity rows live in `user_profiles` (mirror of `auth.users` — user-editable fields only: display_name/email/phone), with RLS so the API role may only read/update the row where `user_id = app.acting_user_id()`; firm-controlled employee fields (`full_legal_name`, `bar_numbers`, `is_admin`) live on `_employees` keyed by composite `(office_id, user_id)`; team/RBAC config lives in `teams`, `team_roles.role_config`, and `team_member_roles`; `entities` is office-scoped legal-party representation attached to matters via `entity_roles`
- Matters use `stage` (allowed: consultation, setup, active, closed, archived — not `open`); other workflow entities use `status` (tasks, invoices, leads)
- Multi-tenant/team integrity via composite FKs: child `(office_id, fk)` → parent `(office_id, pk)`; tenant-scoped parents carry `UNIQUE (office_id, <entity>_id)`; team-owned resources carry `team_id` and must not drift from parent matter/lead/invoice links; `_matter_staff` uses PK `(office_id, matter_id, user_id, matter_role)` and a partial unique index enforcing exactly one `'responsible_attorney'` per active matter
- Soft-delete via `is_deleted` is opt-in only where it adds value, with partial indexes `WHERE NOT is_deleted` (plus `<col> IS NOT NULL` for nullable FK columns) to match active views; `app.audit_log` carries `office_id` for permissioned viewing, is append-only, stores `{col:{from,to}}` diffs, drops `updated_at`/`updated_by` from diffs, skips no-op updates, and `changed_by` prefers `current_setting('app.acting_user_id')` with fallback to row's `updated_by`
- Hono API: use `createMiddleware<AppEnv>` from `hono/factory` for middleware — it propagates env types to `c.set`/`c.get` correctly and is the recommended Hono pattern; `authenticate` is applied globally in `main.ts` and the API is fully auth-gated — any public route (e.g. a healthcheck) must be registered before `app.use(authenticate)`; `created_by`/`updated_by` sourced from `c.get("authUser").id`, never from request body; chain route definitions (`.get().post()`) for RPC-ready type inference; SELECT row types are `Selectable<T>` (not the raw table type `T`); call `.selectAll()` so Kysely infers the row shape; tsconfig `compilerOptions.types` must be `["bun"]`, not `"bun-types"`
- RLS in `app` migrations: policies use `CREATE POLICY … TO service_role` so row filters apply to the API connection role (Postgres `TO public` names the PUBLIC pseudo-role, not “the internet”; `REVOKE` on `app` for `anon`/`authenticated` is still the privilege gate); row checks only apply after `ALTER TABLE … ENABLE ROW LEVEL SECURITY`, and helpers referenced in policy expressions need `GRANT EXECUTE … TO service_role`. Office-scoped policies combine `permitted_office_ids()` with `app.acting_office_id()`: when the office GUC is unset, `SELECT`/`UPDATE` `USING` may span permitted offices; when set, reads and updatable rows are limited to that office; `INSERT`/`UPDATE` `WITH CHECK` requires `office_id = app.acting_office_id()` (writes use a pinned office from `runAsEmployee`). In `LANGUAGE plpgsql` bodies, call `RETURNS void` setup helpers with `PERFORM`, not `SELECT`, or Postgres raises “query has no destination for result data”
