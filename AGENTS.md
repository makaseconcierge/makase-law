## Learned User Preferences

- Uses Bun exclusively â€” not npm, pnpm, or Node; workspace rule enforces this
- Still in planning/prototyping â€” wants a concise plan before schema/logic changes, and it's ok to rewrite migrations wholesale instead of adding new ones
- v0 scope is task management + time tracking (Asana + BigTime replacement); matter staffing (`_matter_staff` â€” not `matter_attorneys`, since paralegals/EAs can also be staffed) is v1
- API connects directly to Postgres via Kysely, not via supabase-js
- Invoice payment allocation modeled as multiple payment rows linked by external_payment_id, not a junction table; `total_amount` is stored and can be overridden via an explicit `adjustment_amount` column, not silently recomputed
- Time entries without a task_id are non-billable, reporting-only (billable/no_charge flags live on tasks, not time_entries)
- Multi-tenant product: firm-specific config (e.g. late_fee_rate) must not have defaults â€” each firm sets its own values
- Pushes back on premature optimization; prefers query simplicity until scale pain is measured
- Values concise DX â€” prefers a single well-considered solution (views, triggers, scopedDb, runAs) over belt-and-suspenders safeguards or app-level wrappers that require type hacks; undelete is just resetting `deleted_at`/`deleted_by` to NULL (audit log preserves history)
- Bun type deps only: `@types/bun` with tsconfig `"types": ["bun"]`; never add `@types/node` even when `Buffer` is needed
- Audit trail is a hard requirement â€” store JSONB diffs only (not full row snapshots); `changed_by` read from `app.acting_user_id` GUC (set by `runAs`) with row's `updated_by` as fallback, so direct-SQL writes still log
- Prefers zod `satisfies` pattern for request validation â€” define schemas once and assert they satisfy `Insertable<DB[...]>` / Patch types, rather than duplicating column lists across SQL and TS

## Learned Workspace Facts

- Bun monorepo: `apps/*` (`client-portal-web-app`, `law-office-web-app` â€” both Vite+React; `api` â€” Bun runtime) plus `packages/*` (`@makase-law/types`, `@makase-law/shared`); root scripts `dev:api`/`dev:client`/`dev:office` (no combined "run all" â€” three terminals)
- All DB tables live in `app` schema, prefixed `_` (e.g. `app._user_profiles`); exposed via `security_invoker = true` views with clean names (`app.user_profiles`) that filter `WHERE deleted_at IS NULL`; Kysely queries the view names unchanged
- `INSTEAD OF DELETE` triggers (`app.prevent_delete`) attached to every view to block hard deletes through views; all `app.*` functions pin `SET search_path = ''`; `anon`/`authenticated` roles fully revoked from `app` (public schema reserved for views when frontend/realtime access is needed); Postgres extensions (e.g. `pg_trgm`) live in a dedicated `extensions` schema, never `public`, to satisfy Supabase advisors
- Kysely query builder with `.withSchema("app")` over Bun's built-in SQL driver; `kysely-codegen --dialect postgres --default-schema app --singularize` generates the `DB` export (not `Database`) into `packages/types/src/dbTypes.ts`; shared types exported from `@makase-law/types`; `pg` is a devDep only for the codegen CLI
- `packages/shared` (`@makase-law/shared`) houses the Kysely `db` client, `runAs`, and all services; consumed by `apps/api` and future cron scripts via `workspace:*`
- `runAs(userId, fn)` wraps writes in a transaction that `SET LOCAL app.acting_user_id`; `set_audit_fields()` BEFORE-trigger on every `_table` writes `created_by`/`updated_by` from that GUC and RAISES if unset â€” every write must run inside `runAs`/`runAsSystem`; auth triggers (`handle_new_auth_user`, `handle_auth_user_updated`) set the GUC to `NEW.id` so auth-originated profile writes self-attribute; system writes run as a fixed SYSTEM user (display name `'SYSTEM'`); applied to non-GET routes only; throws on actor mismatch (one user per request/transaction, never nested)
- Named `{table}_id` PK convention (user_id, office_id, matter_id, entity_id, task_id, invoice_id, etc.) â€” no generic `id`; identity rows live in `_user_profiles` (mirror of `auth.users` â€” user-editable fields only: display_name/email/phone), while firm-controlled fields (`full_legal_name`, `bar_numbers`, `functional_roles`) live on `_employees` keyed by composite `(office_id, user_id)`; `_entities` holds legal-party representation attached to matters via `_entity_roles`
- Matters use `stage` (allowed: consultation, setup, active, closed, archived â€” not `open`); other entities use `status` (tasks, invoices, forms, leads)
- Multi-tenant integrity via composite FKs: child `(fk, office_id)` â†’ parent `(pk, office_id)`; tenant-scoped parents carry `UNIQUE (pk, office_id)` (`*_office_uk`, ordered `(office_id, <entity>_id)`); entities are NOT office-scoped â€” office scoping lives on `entity_roles`, which links entities to matters per office; `_matter_staff` uses composite FKs to both `_matters` and `_employees` (enforcing same office) with PK `(office_id, matter_id, user_id, role)` and a partial unique index enforcing exactly one `'responsible'` per active matter
- Soft-delete via `deleted_at`/`deleted_by` on core tables, with partial indexes `WHERE deleted_at IS NULL` (plus `<col> IS NOT NULL` for nullable FK columns) to match the active views; `app._audit_log` ledger (`table_schema`, `table_name`, `record_pk` JSONB, `op`, `diff` JSONB, `changed_by`, `changed_at`) is append-only with no audit columns on itself; trigger stores `{col:{from,to}}` diffs, drops `updated_at`/`updated_by` from diffs, skips no-op updates, and `changed_by` prefers `current_setting('app.acting_user_id')` with fallback to row's `updated_by`
- Hono API: use `MiddlewareHandler<AppEnv>` for middleware (`createMiddleware<AppEnv>` doesn't propagate env types to `c.set`/`c.get`); `authenticate` is applied globally in `main.ts` and the API is fully auth-gated â€” any public route (e.g. a healthcheck) must be registered before `app.use(authenticate)`; `created_by`/`updated_by` sourced from `c.get("authUser").id`, never from request body; chain route definitions (`.get().post()`) for RPC-ready type inference; SELECT row types are `Selectable<T>` (not the raw table type `T`); call `.selectAll()` so Kysely infers the row shape; tsconfig `compilerOptions.types` must be `["bun"]`, not `"bun-types"`
- Service/route type aliases live in `packages/types/src/makaseLawTypes.ts` (e.g. `Foo = Selectable<DB["foo"]>`, `NewFoo = Insertable<DB["foo"]>`, Patch types for updates) â€” when adding a new table, add the aliases here and import from `@makase-law/types` rather than inlining `Insertable`/`Updateable<DB[...]>` in services; `packages/types/src/remappedDB.ts` exposes `DB` as `RemapViews<_DB>` so `DB["foo"]` (view) resolves to the underlying `_foo` table type, preserving NOT NULL + `Generated<...>` that Postgres drops at the view layer (codegen column overrides must be applied to both `foo.col` and `_foo.col` to survive the remap)
