## Learned User Preferences

- Uses Bun exclusively ā€” not npm, pnpm, or Node; workspace rule enforces this
- Still in planning/prototyping ā€” wants a concise plan before schema/logic changes, and it's ok to rewrite migrations wholesale instead of adding new ones
- v0 product focus is task management + time tracking (Asana + BigTime replacement); forms are not a core table/feature; matter staffing uses `matter_staff`/`matter_role` (not `matter_attorneys`, since paralegals/EAs can be staffed) but staffing workflows can stay later scope
- API connects directly to Postgres via Kysely, not via supabase-js
- Invoice payment allocation modeled as multiple payment rows linked by external_payment_id, not a junction table; `total_amount` is stored and can be overridden via an explicit `adjustment_amount` column, not silently recomputed
- Time entries without a task_id are non-billable, reporting-only (billable/no_charge flags live on tasks, not time_entries); do not store per-entry billable hours on time_entries ā€” task-level billable totals are derived by summing entries for the task and rounding
- Multi-tenant product: firm-specific config (e.g. late_fee_rate) must not have defaults ā€” each firm sets its own values
- Pushes back on premature optimization; prefers query simplicity until scale pain is measured
- Values concise DX ā€” prefers a single well-considered solution (views, triggers, scopedDb, runAs) over belt-and-suspenders safeguards or app-level wrappers that require type hacks; if a workaround seems necessary, stop and present options/recommendations before coding
- Bun type deps only: `@types/bun` with tsconfig `"types": ["bun"]`; never add `@types/node` even when `Buffer` is needed
- Audit trail is a hard requirement ā€” store JSONB diffs only (not full row snapshots); `changed_by` read from `app.acting_user_id` GUC (set by `runAs`) with row's `updated_by` as fallback, so direct-SQL writes still log
- Prefers zod `satisfies` pattern for request validation ā€” define schemas once and assert they satisfy `Insertable<DB[...]>` / Patch types, rather than duplicating column lists across SQL and TS

## Learned Workspace Facts

- Bun monorepo: `apps/*` (`client-portal-web-app`, `law-office-web-app` ā€” both Vite+React; `api` ā€” Bun runtime) plus `packages/*` (`@makase-law/types`, `@makase-law/shared`); root scripts `dev:api`/`dev:client`/`dev:office` (no combined "run all" ā€” three terminals)
- App DB objects live in the `app` schema; soft-deletable tables use underscored base tables plus `security_invoker = true` active views (e.g. `app._matters` -> `app.matters`) and may expose `_all` views for ergonomic joins, while non-soft-deleted tables can be direct clean tables (e.g. `app.offices`, `app.tasks`, `app.invoices`)
- `INSTEAD OF DELETE` triggers (`app.prevent_delete`) attached to every view to block hard deletes through views; all `app.*` functions pin `SET search_path = ''`; `anon`/`authenticated` roles fully revoked from `app` (public schema reserved for views when frontend/realtime access is needed); Postgres extensions (e.g. `pg_trgm`) live in a dedicated `extensions` schema, never `public`, to satisfy Supabase advisors
- Kysely query builder with `.withSchema("app")` over Bun's built-in SQL driver; DB types are generated by Kanel + `kanel-kysely` via `packages/types/kanel.config.cjs` into `packages/types/src/generated`; `@makase-law/types` exports `DB` from `generated/Database` plus generated app/auth aliases; `pg` is a devDep for codegen only; in `.where((eb) => ...)` callbacks, `eb` is `ExpressionBuilder` ā€” build predicates with `eb(column, operator, value)` and `eb.or` / `eb.and`, not `eb.where`
- `packages/shared` (`@makase-law/shared`) houses the Kysely `db` client, `runAs`, and all services; consumed by `apps/api` and future cron scripts via `workspace:*`
- `runAs(userId, fn)` wraps writes in a transaction that `SET LOCAL app.acting_user_id`; `set_audit_fields()` BEFORE-trigger on every `_table` writes `created_by`/`updated_by` from that GUC and RAISES if unset ā€” every write must run inside `runAs`/`runAsSystem`; auth triggers (`handle_new_auth_user`, `handle_auth_user_updated`) set the GUC to `NEW.id` so auth-originated profile writes self-attribute; system writes run as a fixed SYSTEM user (display name `'SYSTEM'`), including bootstrap where SYSTEM creates the office and employee rows (end users do not create offices); applied to non-GET routes only; throws on actor mismatch (one user per request/transaction, never nested)
- Named `{table}_id` PK convention (user_id, office_id, matter_id, entity_id, task_id, invoice_id, etc.) ā€” no generic `id`; identity rows live in `user_profiles` (mirror of `auth.users` ā€” user-editable fields only: display_name/email/phone), while firm-controlled employee fields (`full_legal_name`, `bar_numbers`, `is_admin`) live on `_employees` keyed by composite `(office_id, user_id)`; team/RBAC config lives in `teams`, `team_roles.role_config`, and `team_member_roles`; `entities` is office-scoped legal-party representation attached to matters via `entity_roles`
- Matters use `stage` (allowed: consultation, setup, active, closed, archived ā€” not `open`); other workflow entities use `status` (tasks, invoices, leads)
- Multi-tenant/team integrity via composite FKs: child `(office_id, fk)` ā†’ parent `(office_id, pk)`; tenant-scoped parents carry `UNIQUE (office_id, <entity>_id)`; team-owned resources carry `team_id` and must not drift from parent matter/lead/invoice links; `_matter_staff` uses PK `(office_id, matter_id, user_id, matter_role)` and a partial unique index enforcing exactly one `'responsible_attorney'` per active matter
- Soft-delete via `deleted_at`/`deleted_by` is opt-in only where it adds value, with partial indexes `WHERE deleted_at IS NULL` (plus `<col> IS NOT NULL` for nullable FK columns) to match active views; `app.audit_log` carries `office_id` for permissioned viewing, is append-only, stores `{col:{from,to}}` diffs, drops `updated_at`/`updated_by` from diffs, skips no-op updates, and `changed_by` prefers `current_setting('app.acting_user_id')` with fallback to row's `updated_by`
- Hono API: use `createMiddleware<AppEnv>` from `hono/factory` for middleware ā€” it propagates env types to `c.set`/`c.get` correctly and is the recommended Hono pattern; `authenticate` is applied globally in `main.ts` and the API is fully auth-gated ā€” any public route (e.g. a healthcheck) must be registered before `app.use(authenticate)`; `created_by`/`updated_by` sourced from `c.get("authUser").id`, never from request body; chain route definitions (`.get().post()`) for RPC-ready type inference; SELECT row types are `Selectable<T>` (not the raw table type `T`); call `.selectAll()` so Kysely infers the row shape; tsconfig `compilerOptions.types` must be `["bun"]`, not `"bun-types"`
- Service/route type aliases live in `packages/types/src/makase-types.ts` and import Kanel-generated table aliases directly (e.g. `Offices`, `NewOffices`); `New*`/`*Patch` aliases omit trigger-owned audit and soft-delete columns, and services should import these readable aliases from `@makase-law/types` rather than raw `Insertable`/`Updateable` shapes
